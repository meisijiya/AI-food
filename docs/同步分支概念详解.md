# 🔄 Git 分支同步：到底同步了什么？

> 本文档澄清 Git Worktree 并行开发中最常见的误解——**"同步分支"不等于"两个分支一模一样"**。
>
> 核心问题：同步分支，是指主分支和副分支的状态一样吗？同步后，沙盒的状态和主分支状态一样吗？

---

## 🎯 一句话答案

**"同步"不是"两个分支一模一样"**，而是让两个分支**从一个共同的最新点开始分叉**。

同步后：

```
同步前（feature 落后 master 5 个提交）：
master:     A---B---C---D---E---F
                 \
feature:         X---Y          ← 停在 C 之后

rebase 后（feature 接到了最新 master 后面）：
master:     A---B---C---D---E---F
                                \
feature:                        X'---Y'   ← 基于 F，X/Y 内容保留
```

> ✅ **沙盒目录里能看到 master 的 F，也能看到自己的 X、Y**。两边合在一起就是 feature 分支的全部历史。

---

## 🤔 那到底"一样"在哪？

| 维度 | 同步前 | 同步后 |
|---|---|---|
| feature 包含 master 的提交？ | ❌ 只有 B、C，缺 D、E、F | ✅ 全部 A~F |
| feature 有自己的提交？ | ✅ X、Y | ✅ X'、Y'（内容相同，hash 可能变） |
| 工作目录文件状态 | 只有 B、C、X、Y | master 全部 + X、Y 的混合 |
| 跟 master 的"分叉点" | 旧的（C） | 新的（F） |

---

## ⚠️ 容易踩的 3 个误区

### 误区 1："同步后沙盒和 master 一样"

**❌ 不对。** 如果你已经在 feature 上写了 X、Y 两个提交，**这些提交不会消失**。同步只是把 X、Y 重新"贴"到最新 master 后面。

### 误区 2："同步会丢掉我写的代码"

**❌ 不对。** `git rebase master` 的设计就是**保留你的提交**，只是重新计算它们的"父节点"。文件内容一字不差，只是 commit hash 可能变（因为父变了）。

### 误区 3："那我直接 reset 到 master 不就同步了？"

**❌ 不对。** reset 是把 feature 分支**整个扔掉**，回到干净的 master——适用于"这功能砍掉不要了"的场景。**rebase 才是同步**——保留你的工作，只是基于最新 master。

---

## 💡 极端情况：刚开的新分支

如果你刚 `git worktree add -b feature/login` 开了一个新分支，**还没写任何东西**就 rebase，那确实会变成"和 master 一样"——因为没东西可保留嘛。

---

## 🎨 图解：同步前后对比

### 场景 1：Feature 有自己的提交

```
【同步前】
master:     A---B---C---D---E---F
                 \
feature:         X---Y              ← 基于 C，落后 master 3 个提交

【rebase 后】
master:     A---B---C---D---E---F
                                \
feature:                        X'---Y'   ← 基于 F，你的 X/Y 还在
```

### 场景 2：Feature 还没写任何东西

```
【同步前】
master:     A---B---C---D---E---F
                 \
feature:         (空)             ← 基于 C，落后 master 3 个提交

【rebase 后】
master:     A---B---C---D---E---F
                            \
feature:                    (空)         ← 基于 F，内容 = master
```

### 场景 3：Merge 同步（保留分叉历史）

```
【同步前】
master:     A---B---C---D---E---F
                 \
feature:         X---Y

【merge 后】
master:     A---B---C---D---E---F
                 \                   \
feature:         X---Y-----------M   ← 多了一个 merge commit M
```

---

## 🔧 三种"同步"姿势的本质区别

| 姿势 | 本质 | 你的提交保留？ | 历史形态 |
|---|---|---|---|
| **rebase** | 把你的提交"摘下来"接到新 master 后面 | ✅ 保留，hash 可能变 | 直线型（干净） |
| **merge** | 把 master 合到你的分支，多一个 merge commit | ✅ 保留，hash 不变 | 分叉型（真实） |
| **reset** | 把你的提交扔掉，回到干净 master | ❌ 丢弃 | 单线型 |

---

## ✅ 一句话总结

> **同步 = 把 master 的新提交"搬"到 feature 前面，让 feature 接住它们。**
> **同步 ≠ 让 feature 变成 master。**
> **同步后沙盒 = 最新 master + 你的功能提交。**

---

## 💧 最直观的理解（水的比喻）

> 把 master 当成一条流动的河，你的 feature 分支是个支流。同步 = 把支流的入口往上移到最新河段。你的水（提交）还在，只是源头更新了。

---

## 📋 速查表

| 你的问题 | 答案 |
|---|---|
| 同步 = 两边一样？ | ❌ 不一样，是"接住" |
| 同步会丢我的代码？ | ❌ 不会，rebase 保留内容 |
| 同步后沙盒 = master？ | ⚠️ 只有空分支才相等 |
| 同步 = reset？ | ❌ reset 会丢，rebase 才叫同步 |
| 想砍掉 feature 重新来？ | ✅ 用 reset，不要 rebase |

---

## 🎯 实战建议

1. **日常同步** → `git rebase master`（干净历史）
2. **协作场景** → `git merge master`（保留真实分叉）
3. **砍掉重来** → OpenCode UI reset 或 `git reset --hard origin/master`
4. **避免大冲突** → 小步 rebase，常同步，别让 feature 落后太多
